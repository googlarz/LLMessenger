# LLMessenger Performance Analysis — 2026-06-12

Static analysis (code-evidence based; no profiler run — the app was not executed). Every finding cites file:line.
Cost figures are estimates derived from operation type × call frequency; where marked *estimated* they should be
confirmed with Instruments (Time Profiler + SwiftUI template) before optimizing further.

**Architecture context for frequency math:** `AppState` is one `ObservableObject` with 8 `@Published` properties
(`UI/AppState.swift:113-126`). Any change to *any* of them (poll health update, brief refresh, card filed,
countdown poll date) re-evaluates the `body` of every view observing it via `@EnvironmentObject` — which is
nearly every view in the panel. So "per render" below means "per poll cycle, per card click, per hover, per
brief switch", not "once".

## Ranked findings

### P1 — Synchronous DB reads inside `BriefCardView.stampRow` render path (N+1, per card, per render)

**Where:** `LLMessenger/UI/BriefCardView.swift:128` (render path), `:117` (button action — acceptable).
Backed by `UI/AppState.swift:360-362` → `Core/Brief/BriefRepository.swift:599-603` → `database.dbQueue.read`.

**Why it's hot (frequency × cost):**
- Line 128 sits inside the `label:` closure of the conversation-name button — evaluated on **every render of every card**.
- A brief with 10 cards = 10 synchronous `dbQueue.read` calls per render pass of `BriefProseView`.
- Each card has its own `@State hovering` (`BriefCardView.swift:70,102`); **every mouse-enter/leave re-renders that card** → 1 DB read per hover transition, on the main thread.
- Worst case: `DatabaseQueue` serializes reads behind writes. If a poll-cycle write transaction is committing
  (storing hundreds of messages + 3 FTS5 triggers each, see P4), the main thread **blocks until the write
  finishes** — a visible UI hang during hover/scroll. Cost per read: ~0.1ms uncontended (PK lookup), but
  10-100ms+ when queued behind a bulk insert.

**Fix (minimal):** Fetch all conversation contexts for the brief's cards **once** when the brief loads (single
`WHERE (service, conversationId) IN (...)` query, off-main via `Task.detached` like `refreshBriefs()`), store in a
`[String: ConversationContext]` dictionary on `ChatViewModel` or a small `@State` in `BriefProseView`, and have
`stampRow` do a dictionary lookup. Invalidate on `saveConversationContext`.

### P2 — `BriefProseView.parsedJSON` decodes the full brief JSON ~8-10× per body evaluation

**Where:** `LLMessenger/UI/BriefProseView.swift:35-45` (the computed property: 2 regex
`replacingOccurrences` + `JSONDecoder()` alloc + full decode per access).

**Access chain per single `body` evaluation:**

| Accessor | Line | Calls to `parsedJSON` |
|---|---|---|
| `services` | 47-52 | 1 |
| `body` direct | 91, 97 | 2 |
| `cardCounts` (via `filterLine` → `filterItem` ×N services) | 58-64, 236, 239 | 1 per filter item |
| `numberedVisibleCards` (via `highPriorityCards` :74, `otherCards` :78, `markAllRow` :185, keyboard `.background` :135) | 66-72 | 4 |

Total: **~8-10 full JSON decodes + 16-20 regex passes per render** of the brief body. The opening summary for a
busy brief is multi-KB (cards × headline+summary+actions+quotes); a 10-20KB decode is roughly 0.1-0.5ms, so
~1-5ms of pure redundant CPU per render — *every* poll tick, hover-adjacent invalidation, card filing, or brief
switch. This alone can eat a 16ms frame budget on animation frames (the staggered fade-up at :172 animates while
re-evaluating).

**Identical pattern:** `UI/ChatPanelView.swift:20-54` — `headerStats` re-decodes the same JSON (regex ×2 +
decode, :32-48) plus `decodedStringArray` (:26 → :169-175) on every render of `ChatPanelView`; called at :62.

**Fix (minimal):** Decode once per brief. Either:
- `let parsed: BriefJSON?` computed in `ChatPanelView` and passed *down* to `BriefProseView`/`BriefHeaderView` as a stored property (views are recreated cheaply; the decode happens once per parent render instead of 10× per child render), or
- cache keyed by `brief.id` in `ChatViewModel.loadBrief` (already runs per brief switch, `ChatPanelView.swift:143-146`) and expose `@Published var parsedBrief: BriefJSON?`.
The second removes the decode from the render path entirely.

### P3 — Main-thread synchronous DB work in `BriefListView` and per-keystroke FTS search on MainActor

**Where:**
- `LLMessenger/UI/BriefListView.swift:198-206` `refreshNeedsReply()` → `AppState.fetchNeedsReplyCards()`
  (`UI/AppState.swift:331-333`) → `fetchRecentHighPriorityCards` — a **synchronous main-thread read** triggered
  `.onAppear` (:191), on **every `briefs` change** (:192) and on **every `handledCardKeys` change** (:193 — i.e.
  every single FILE/REOPEN click re-runs the query).
- `LLMessenger/UI/BriefListView.swift:283-297` `performSearch` uses `Task { }` which **inherits MainActor** from
  the SwiftUI context; `searchMessages` + `searchBriefs` (both synchronous `dbQueue.read`, FTS5 MATCH over the
  whole archive) therefore run **on the main thread, once per keystroke, with no debounce**. `searchTask?.cancel()`
  (:276) cannot help: cancellation is only checked *after* both synchronous searches complete (:287).
- Same pattern: `UI/ConversationTimelineView.swift:70-74` (`loadEntries` sync read in `.task` on MainActor) and
  `UI/BriefCardView.swift:436-445` (`loadEvidence` sync read in `.task`), `UI/ContactProfileView.swift:65-84`
  (see P7).

**Why it's hot:** typing "hello" = 5 main-thread FTS scans back-to-back. On a 50k-message archive each MATCH +
rank + snippet is several ms; queued behind any in-flight write, far more. The needs-reply query additionally
suffers the missing index (P5).

**Fix (minimal):** `Task.detached(priority: .userInitiated)` for the search body (the repository is already used
from detached tasks in `AppState.refreshBriefs`), plus a 200-300ms debounce (`try await Task.sleep` before
querying, cancellation then actually works). Move `refreshNeedsReply` to the same detached pattern; for the
`handledCardKeys` trigger, filter the *already fetched* array locally instead of re-querying (handled-filtering
is already done in Swift at :85-87 — the DB re-query at :193 adds nothing).

### P4 — `PollEngine` is `@MainActor`: bulk message insert + health writes run on the main thread

**Where:** `LLMessenger/Core/PollEngine.swift:4` (`@MainActor` on the whole class).
- `store(result:service:)` :246-268 — synchronous `dbQueue.write` inserting every fetched message; each insert
  also fires the 3 FTS5 sync triggers (`Core/Store/AppDatabase.swift:178-197`). First poll fetches a 48h window
  (`PollEngine.swift:228`) — potentially thousands of rows inserted in one main-thread transaction.
- `writeHealth` :270-300 — synchronous `dbQueue.write` per service per poll, **even when status is unchanged**
  (ok→ok), each followed by a `NotificationCenter` post (:293) that makes the Settings tab re-read all health
  rows. 4 services × every poll cycle = 4 redundant write transactions + 4 UI invalidations per cycle. It also
  performs a read *inside* the write when `updateLastCheck == false` (:279).
- Because all UI DB reads share the same `DatabaseQueue`, these main-thread writes are exactly what P1/P3 reads
  queue behind.

**Timer details:** `scheduleTimer` :174-192 creates one repeating `Timer` per service with **no `tolerance`**
— the OS cannot coalesce wakeups; with 4 services at independent 900s phases the app wakes ~4× more often than
needed (energy, not latency). Fetch windows themselves are CLEAN: since-last-check anchoring with a 48h
first-run cap (:225-243), and `readLastCheck` is correctly off-main via `Task.detached` (:202-208).

**Fix (minimal):**
1. Move `store` and `writeHealth` bodies into `Task.detached` / a non-MainActor helper (the `AppDatabase` is
   already `@unchecked Sendable`, `AppDatabase.swift:5`).
2. Skip `writeHealth` + notification when `(status, error)` equals the previous in-memory value.
3. `timer.tolerance = interval * 0.1`.

### P5 — Missing index for the needs-reply query (`briefCards.priority` + `createdAt`)

**Where:** `Core/Brief/BriefRepository.swift:465-485`:

```sql
SELECT bc.*, b.createdAt AS briefCreatedAt
FROM briefCards bc JOIN briefs b ON bc.briefId = b.id
WHERE bc.priority = 'high'
ORDER BY bc.createdAt DESC LIMIT ?
```

`Core/Store/AppDatabase.swift` indexes `briefCards` only on `briefId` (:147) and `(service, conversationId)`
(:148). Neither covers `priority` or `createdAt` → **full table scan + sort of all briefCards ever created** on
every execution — and P3 shows this executes on every FILE click and every brief refresh, on the main thread.
briefCards grows without bound (~5-20 rows per brief, ~24+ briefs/day → tens of thousands of rows within months).

**Fix (minimal):** one migration:
`CREATE INDEX briefCards_on_priority_createdAt ON briefCards(priority, createdAt DESC)` — turns it into an index
range scan, no sort.

**Other queries audited against the schema:**

| Query | Index used | Verdict |
|---|---|---|
| `fetchMessages(service:since:)` :217 | `messages_on_service_timestamp` (v12, :284) | CLEAN |
| `fetchMessages(forBriefID:)` :227 | `messages_on_briefId` (v2, :79) | CLEAN |
| `fetchRecentContextMessages` :236 | `messages_on_service_conversation` (v11, :278) | CLEAN |
| conversation timeline join :499-514 | `briefCards_on_service_conversation` + briefs PK | CLEAN |
| `fetchSourcesWithMessages` :538/:551 + `attachMessages` :570-586 | batched, no N+1 | CLEAN (explicitly fixed) |
| `storeMessages` :151-214 | bulk pre-read, chunked at 500 params | CLEAN (explicitly fixed) |
| `fetchAllBriefs` :261 | `briefs_on_createdAt` (v12, :290) | index OK, but **unbounded** — see P8 |
| `recentEpisodicSummaries` :132 | `services LIKE '%...%'` non-sargable | minor — briefs table is small-ish, has LIMIT |
| `fetchUnreadCount` :394 | no index on `briefs.status` | minor — small table |
| `fetchUnattachedMessages` :19 | `messages_on_briefId` covers `IS NULL` | CLEAN |

### P6 — `DateFormatter()` allocated inside view render paths — 9 sites in UI

`DateFormatter` construction + `dateFormat` set costs ~50-500µs (locale/calendar/ICU setup) and churns the heap.
Per-row usages multiply it.

| Site | File:line | Frequency |
|---|---|---|
| `BriefRowView.timeLabel` | `UI/BriefListView.swift:509` | **per visible row, per list render** (every hover of any row animates → re-render) |
| `BriefProseView.timeStr` | `UI/BriefProseView.swift:371` | per quote row in fallback view |
| `BriefCardEvidenceView.timeStr` | `UI/BriefCardView.swift:457` | per citation row when evidence expanded |
| `TimelineEntryRow.dateStr` | `UI/ConversationTimelineView.swift:167` | per timeline row |
| `MainChromeBar.currentBriefLabel` | `UI/MainChromeBar.swift:207` | per chrome-bar render (every AppState change) |
| `BriefHeaderView.timeRange` | `UI/BriefHeaderView.swift:160` | per header render |
| `TaskListView.briefDateLabel` | `UI/TaskListView.swift:78` | per task row |
| `BriefListView.dateRangeLabel` | `UI/BriefListView.swift:301` | only when date filter active — minor |
| `PrivacySettingsTab` | `UI/Settings/PrivacySettingsTab.swift:135` | settings only — minor |

Also `TimelineEntryRow.actionItems` (`UI/ConversationTimelineView.swift:83-87`) JSON-decodes per render per row.

**Counter-example already in the codebase:** `BriefListGrouper.mediumDateFormatter`
(`UI/AppState.swift:95-100`) is a cached `static let` — the correct pattern, just not applied elsewhere.

**Fix (minimal):** one `enum Formatters { static let hhmm: DateFormatter = ... }` (formatters are thread-safe
since macOS 10.9 for reading) and replace the 9 alloc sites. ~30 lines total.

### P7 — `ContactProfileView` decodes every brief in the archive on the main actor

**Where:** `UI/ContactProfileView.swift:65-84`: `loadProfile()` (`.task`, MainActor) calls `fetchAllBriefs()`
(full table, sync read) then **JSON-decodes every brief's `openingSummary`** until 5 matching cards are found.
For a conversation with little history this walks and decodes the *entire archive* (months × 24 briefs/day ×
multi-KB decode each) on the main thread when the profile sheet opens.

**Fix (minimal):** the data already exists in relational form — `briefCards` has `(service, conversationId)`
indexed. Use `fetchConversationTimeline` (`BriefRepository.swift:499`) with a `LIMIT 5` variant instead of
decoding JSON blobs; run it detached. Removes both the unbounded scan and all decoding.

### P8 — Unbounded `fetchAllBriefs` into memory on every refresh

**Where:** `UI/AppState.swift:221` (`refreshBriefs` → `fetchAllBriefs`), re-run after *every* mark-as-open,
pin, archive, snooze, poll cycle (`AppState.swift:208,278,307,316,325`; `AppDelegate.swift:111,151,186,281`).
`briefs` (every brief ever, each carrying its full multi-KB `openingSummary` text) is then:
- held in memory permanently,
- re-grouped/sorted by `briefGroups` per list render (`AppState.swift:175-177`, `BriefListView.swift:26-33`),
- copied into the menu bar via `setBriefs` on every change (`AppDelegate.swift:113,153,188,248,298`).

At ~24 briefs/day this is fine for weeks and degrades quietly over months (memory + O(n log n) regroup per
render + full-array diffing).

**Fix (minimal):** add a `LIMIT` (e.g. 500, archived excluded by default) or a `since:` window to the default
`fetchAllBriefs` call; the date-filter and archived sections can fetch on demand
(`fetchBriefs(from:to:)` already exists, `BriefRepository.swift:306`).

### P9 — Launch path: synchronous DB migration + Keychain reads before first paint

**Where:** `AppDelegate.applicationDidFinishLaunching` (`AppDelegate.swift:27-397`), all on the main thread
before the run loop turns:

1. `try AppDatabase()` :32 — DB open + 16 migrations checked; on first launch after the v6 upgrade the FTS5
   backfill (`AppDatabase.swift:172-175, 207-210`) rewrites the whole messages/briefs tables (one-time, but can
   be seconds on a large DB).
2. `resolvedProvider()` :35 → `loadLLMKey` (`SettingsRepository.swift:33`) — **synchronous Keychain
   (Security.framework) call**, typically 5-50ms, occasionally much worse.
3. `registerAdapter` ×4 :332-334 — each does a sync `dbQueue.read` for `ServiceConfig` (:565) plus more Keychain
   reads (`loadSignalAccount` :581, `loadTelegramCredentials` via :659) and `SlackWorkspaceStore.load()` file I/O (:611).
4. CLEAN: `engine.start()` is wrapped in a `Task` (:337-340) and polls concurrently; `state.refreshBriefs()`
   (:374) is `Task.detached`.

Estimated 50-300ms of main-thread work before the menu-bar icon responds; acceptable for a menu-bar app but
easily halved by deferring (2) and (3) into the same startup `Task` as `engine.start()`.

**Correctness observation (not perf):** `AppDelegate.swift:374-376` — `refreshBriefs()` is async/detached, but
`state.briefs.isEmpty` and `menuBar.setBriefs(state.briefs)` are read **immediately after**, before the fetch
lands. On every launch the state is computed from an empty array (`briefGenerationState = .noNewMessages`,
empty menu) until the first `onBriefsChanged` fires. Flag for a separate fix.

### P10 — Smaller render-path items (grouped)

- `ChatPanelView.briefMessages` (`UI/ChatPanelView.swift:8-12`) and `aiItems` (:14-18) re-filter
  `threadItems` on each access; `briefMessages` is accessed by `headerStats` (:21) and `body` (:80) — 3-4 array
  passes per render. Trivial to compute once in `body` via `let`.
- `AppState.markAllHandled` (`UI/AppState.swift:265-273`) decodes the brief JSON again, then calls
  `markCardHandled` per card — each call serializes the whole `handledCardKeys` set to `UserDefaults`
  (:251-254): O(n²) writes for "FILE ALL". Batch: one insert loop + one `UserDefaults.set`.
- `BriefProseView.summaryBlock` (`UI/BriefProseView.swift:300-313`) re-parses markdown
  (`AttributedString(markdown:)`) per render — fallback path only.
- `AppDelegate.highPriorityNotification` (`AppDelegate.swift:629-643`) decodes brief JSON — but only per
  notification, event-driven: CLEAN.
- Per-second `TimelineView(.periodic by: 1)` countdowns (`UI/ChatPanelView.swift:185`,
  `UI/BriefListView.swift:387`) are correctly scoped so only the text re-renders: CLEAN.

## PRD section 17 / performance acceptance tests — DISCREPANCY

**The PRD does not contain a section 17.** `docs/PRD.md` ends at section 10 ("Open Questions", line 624);
sections are 1-10 plus roadmap subsections. No performance requirements, latency budgets, or acceptance tests
are defined anywhere in `docs/` (grep for performance/latency/p95/launch time/frame across `docs/` returns only
unrelated hits).

**Test suite status:** `LLMessengerTests/` contains **zero performance tests** — no `measure {}` blocks, no
`XCTMetric`/`XCTMeasureOptions` usage anywhere (47 test files checked). All timing-related hits are functional
(timeouts, timestamps).

Missing performance tests worth adding (derived from this analysis, since no PRD list exists):

1. `measure {}` on decoding a representative `BriefJSON` and on `BriefListGrouper.group` with 1k briefs.
2. `measure {}` on `fetchRecentHighPriorityCards` with 50k seeded briefCards (guards the P5 index).
3. `measure {}` on `searchMessages` with a 50k-message FTS corpus (guards P3 regressions).
4. A main-thread-blocking assertion around `PollEngine.store` with a 1k-message batch (guards P4).
5. Launch-path budget test: time `AppDatabase()` + migrations against a seeded on-disk DB.

If a section 17 was intended to exist, it needs to be written first; the five tests above are a reasonable
starting acceptance set.

## Top-5 summary (by user-perceived impact)

1. **P1** `BriefCardView.swift:128` — sync DB read per card per render (incl. every hover); can block behind poll writes → visible hangs.
2. **P2** `BriefProseView.swift:35-45` + `ChatPanelView.swift:20-54` — brief JSON decoded ~10×/render; decode once per brief.
3. **P3** `BriefListView.swift:283-297, 198-206` — per-keystroke FTS search and needs-reply query synchronous on MainActor, no debounce.
4. **P4** `PollEngine.swift:4,246-300` — bulk message insert + redundant per-poll health writes on the main thread; no timer tolerance.
5. **P5** `BriefRepository.swift:465-485` — needs-reply query full-scans `briefCards` (no priority/createdAt index), executed per FILE click.
